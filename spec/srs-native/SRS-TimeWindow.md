# SRS — Native Time Window Editing (Explicit Window, Ports, Clipping, Losslessness)

Native-specific requirements for the explicit, user-controlled time window on the PowerPoint on-slide Gantt editor (v2.7.x program). Feature tag: WIN (time window). The window is edited by dragging arrow ports at the left/right ends of the timescale header; out-of-window elements clip or hide losslessly; the chart rescales to the new window on drop (two-phase render).

Traces up to: `docs/time-window-plan.md` (review-amended design, findings C1–C3/M1–M5/m1–m10), `PLAN.md` (Phase 12, v2.7.0–v2.7.4), `../data-model.md`, `../schema/document.schema.json`, `SRS-InteractionConventions.md` (SR-IXC-03/04/07/12), `SRS-InteractionSmoothness.md` (SR-SMO-01/02/06/08).

Reference impl: `native/PowerPlannerAddin/` (GanttScene.h projection + clip stage, GanttBuilder.cpp UpdateGantt/ReconcileChartRoot/ReflowFromSlide, Overlay.cpp gesture + preview paint, GanttHitTest.*, GanttOps.*, GanttJson.cpp), `native/tools/harness_driver.py` + `native/tools/scenarios/trace_window_*.json`.

## Explicit Window & Persistence

| ID | Requirement | Rationale | Verification | Trace |
|----|-------------|-----------|--------------|-------|
| SR-WIN-01 | The document shall carry optional `windowStart` / `windowEnd` (ISO `YYYY-MM-DD`); when absent, the visible range shall be derived exactly as today (`[min(dates) − 5%pad .. max(dates) + 5%pad]`), and the canonical JSON shall omit empty window fields. | Full backward compatibility: every existing PP_DOC tag, fixture, and golden must be untouched by the feature. | ops-test JSON round-trip (set/clear/omit-empty); build-conformance fixtures byte-identical (no fixture sets a window). | v2.7.1 (W1); plan §1.1/§2.1 |
| SR-WIN-02 | An explicit window shall persist in PP_DOC and round-trip unchanged through save, Import from slide, and Repair layout. | The window is a first-class document setting, not a session view state. | trace_window_repair_lossless (PP_DOC window fields identical after Repair layout + resize); trace_window_clip_rerender PP_DOC assertions. | v2.7.1/v2.7.3; plan §1.1 |
| SR-WIN-03 | `SetTimeWindow` / `ClearTimeWindow` shall be pure GanttOps operations that reject malformed dates, enforce `end ≥ start + 1 visible scale unit` and the scale-dependent maximum span, and store exact (unsnapped) dates — snapping is a gesture concern only. | Model-level guards keep every caller (gesture, settings, future API) safe; storage stays precise. | ops-test guard cases (malformed, inverted, sub-unit span, over-max span, snap-free storage). | v2.7.1 (W1); plan §2.1 |
| SR-WIN-04 | The first port drag on an auto-fit chart shall materialize the derived window — including the current 5% pad — as the explicit baseline, producing zero geometry delta at materialization (ptPerDay unchanged). | D2: the only choice with no visible jump when the gesture starts; PP_PROJ carries no maxDay, so windowEnd is reconstructed from doc extents + pad. | trace_window_edge_drag zero-geometry-delta assertion (pre vs materialized dump + pixel hash). | v2.7.2/v2.7.3; D2; plan risk 4 |

## Port Affordances

| ID | Requirement | Rationale | Verification | Trace |
|----|-------------|-----------|--------------|-------|
| SR-WIN-05 | Two theme-token-painted arrow ports (◀ / ▶) shall appear at the left and right ends of the timescale header band, painted and hit-testable only while the cursor is over the header band or a window drag is active, with paint updates on hover transitions only. | Hover-gated affordances keep idle chrome calm (IDLESTABLE) while making the gesture discoverable in place (SR-IXC-13 family). | trace_window_edge_drag (ports visible on header hover); IDLESTABLE stage stays paint-free; full drag-suite regression. | v2.7.2 (W2); review m10 |

Shrinking marker bands would regress existing marker selection. | Full drag-suite re-run (link-port + marker traces green); trace_window_edge_drag clicks derived from dump rects only. | v2.7.2 (W2); review m3, plan risk 3 |
| SR-WIN-07 | The overlay dump shall expose `windowStart`, `windowEnd`, `windowDragActive`, `windowPillText`, `windowPortLRect`, `windowPortRRect` as ground truth for harness profiles. | Profiles must click exactly where the overlay hit-tests (U4/U5 lesson); asserted state must come from the overlay, not screenshots. | trace_window_edge_drag consumes dump rects and asserts the drag-state fields. | v2.7.2 (W2); plan §2.3 |

## Drag Semantics & Two-Phase Render

| ID | Requirement | Rationale | Verification | Trace |
|----|-------------|-----------|--------------|-------|
| SR-WIN-08 | Window edge drags shall snap the candidate edge to the smallest visible scale unit (day at day scale, week at week scale, …), active during live preview and at commit. | Consistent with the marker snap rule (SR-IXC-04/12); grid-aligned windows match the visible axis. | trace_window_edge_drag snapped-candidate assertion in the mid-gesture dump. | v2.7.2 (W2); SR-IXC-12 |
| SR-WIN-09 | During a window drag a live pill shall show the candidate window as two ISO dates plus the span delta (e.g. `2026-06-01 → 2026-08-10 (+7d)`), updating on every mouse move. | Live-preview convention SR-IXC-03: the user sees the exact outcome before committing. | trace_window_edge_drag: `windowPillText` contains two ISO dates mid-gesture. | v2.7.2 (W2); SR-IXC-03 |
| SR-WIN-10 | Drag clamps shall enforce: minimum span of 1 visible scale unit; a scale-dependent maximum span chosen so the emitted axis prim count stays bounded (AXIS_MAJOR ticks ≤ ~130 at the current scale); and neither port may cross the other edge. | A flat 20-year cap at week scale would emit ~240 uncapped AXIS_MAJOR shapes and blow the commit budget; degenerate windows must be unreachable. | ops-test clamp cases (min/max/cross); trace_window_edge_drag over-drag step stops at the clamp. | v2.7.1/v2.7.2; review m5 |
| SR-WIN-11 | During a window drag only the overlay shall repaint: an opaque token-painted axis preview strip covering the header band shows the candidate window's tiers/ticks, and no PowerPoint shape or tag shall be written per mouse move. | The user-specified two-phase design: shapes are touched exactly once, on release; the preview keeps the feel responsive. | trace_window_edge_drag mid-gesture: header-band pixel-diff vs pre AND shape geometry/tags unchanged; IDLESTABLE stays paint-free. | v2.7.2 (W2); plan §1.2/§2.3 |
| SR-WIN-12 | Esc or capture loss mid-drag shall cancel the gesture: the window is unchanged, and the axis preview and pill are cleared with an immediate repaint (CLEAR-MUST-REPAINT). | Uniform cancel semantics (SR-IXC-07); stale preview pixels were a session-learned failure mode. | trace_window_edge_drag Esc-cancel steps (preview gone, window fields unchanged). | v2.7.2/v2.7.3; SR-IXC-07; review m9 |
| SR-WIN-13 | On release the commit shall read only values snapshotted at WM_LBUTTONUP (stale-global rule) and dispatch `SetTimeWindow(start, end)` through the single RebuildChart choke point. | Two U5 bugs came from commits reading live globals wiped by the capture-release path. | trace_window_clip_rerender / trace_window_commit_latency commit steps; code review checkpoint in the W3 dispatch prompt. | v2.7.3 (W3); plan §2.3 |

## Clip Semantics

| ID | Requirement | Rationale | Verification | Trace |
|----|-------------|-----------|--------------|-------|
| SR-WIN-14 | Elements fully outside the window shall emit no shapes: task bar + progress + labels, milestone diamond + label, marker line + chip, and their anchored notes. | Hiding is a render filter; absent shapes are also unselectable and unclickable by construction. | ops-test clip cases (prim absence per kind); trace_window_clip_rerender bar-absent assertion. | v2.7.1/v2.7.3; plan §1.3 |
| SR-WIN-15 | Elements straddling a window edge shall be truncated flat at the boundary and carry a small NON-INTERACTIVE arrow-port continuation glyph at the cut edge (same visual family as the window ports; no hit zone), with progress fill clipped to the bar while its fraction is computed on the TRUE task span. | D1 (user-decided): a truncated bar must be distinguishable from one that genuinely starts/ends there, without adding a competing interactive affordance. | ops-test straddler cases (clippedL/clippedR flags, glyph prim present, no hit-zone entry); clip unit tests incl. tiny clipped widths. | v2.7.1 (W1); D1; plan §1.3/risk 5 |
| SR-WIN-16 | Label prims shall clamp or drop with their owning element and shall never overhang the window edge; the union scene bounds shall be invariant under window changes. | Overhanging milestone/task/marker labels drift the CHART_ROOT bbox, triggering frame re-fit (axis-independent rescale) and the ReflowFromSlide hazard. | ops-test bounds-invariance case (union bounds equal across window changes with overflowing labels near the edge). | v2.7.1 (W1); review M1 |
| SR-WIN-17 | Dependency elbows and brackets shall clip or hide as units: an elbow (all three segments + arrowhead) is hidden when either endpoint is hidden and unit-clipped, never per-segment fragmented, when an endpoint is clipped-but-visible; a bracket's five prims clip together. | Generic per-prim truncation produces dangling elbow stubs and tickless bracket fragments. | ops-test DEP/bracket unit-clip cases; trace_window_clip_rerender deps-hidden assertion. | v2.7.1/v2.7.3; review M3, m8 |
| SR-WIN-18 | Anchored notes shall follow their anchor's clip state: hidden anchor hides the note; a clipped-but-visible anchor keeps the note positioned at the clipped anchor geometry. | Notes derive position from anchor layout; orphaned notes floating over a hidden anchor would be unexplainable. | ops-test anchored-note clip cases. | v2.7.1 (W1); review m8 |
| SR-WIN-19 | Rail prims (row labels, rail dots and rail task labels, dividers) and axis prims shall be exempt from the clip filter: every document row shall render in the left rail regardless of whether it has visible bars. | User requirement: the rail is the complete inventory; the axis defines the window itself. | trace_window_clip_rerender rail-row-present assertion; ops-test exemption case. | v2.7.1/v2.7.3; plan §1.3 |

## Losslessness & Geometry Truth

| ID | Requirement | Rationale | Verification | Trace |
|----|-------------|-----------|--------------|-------|
| SR-WIN-20 | Clipping and hiding shall be a pure scene-builder filter over the unchanged document: moving the window back over hidden/clipped elements shall re-emit them bit-identical (dates, color, percent, label placement, dependencies, geometry). | The window is a view, never an edit; this is the feature's core contract. | trace_window_clip_rerender: shrink → expand → geometry/props identical to the pre capture (dump + pixel hash). | v2.7.3 (W3); plan §1.3 |
| SR-WIN-21 | ReflowFromSlide (and any other geometry back-projection) shall never derive document dates from clipped or hidden shapes; Repair layout and chart resize on a chart with clipped elements shall leave PP_DOC dates byte-identical. | Review C1: back-projecting a clipped bar's Left/Width rewrites its true dates to the clipped span — silent data corruption reachable from the commit path itself (FitChartRootToFrame → ReflowFromSlide). | trace_window_repair_lossless (Repair layout + resize on clipped fixture; PP_DOC date fields byte-identical). | v2.7.1 (W1); review C1 |
| SR-WIN-22 | Under an explicit window, px/day and anchor-day derivations (drag scale, empty-cell create, create-at-point) shall prefer the PP_PROJ projection or skip clipped tasks; a clipped bar's rect shall never be paired with its true day span or start date. | Review C2: snapshot-rect ÷ true-span math and rect.left ↔ task->start pairing are falsified by clipped bars, shifting every create/drag on the chart. | Clipped-chart create/drag steps in trace_window_clip_rerender (created task lands on the clicked day); drag-suite regression on a clipped fixture. | v2.7.3 (W3); review C2 |

## Projection, Caching & Structural Classification

| ID | Requirement | Rationale | Verification | Trace |
|----|-------------|-----------|--------------|-------|
| SR-WIN-23 | When an explicit window is set, the projection window shall be `windowStart..windowEnd` verbatim with no pad; `DocDatesFitPaddedWindow` shall not be consulted (elements legitimately live outside), and scene-cache validity shall key on window equality, with both `SetTimeWindow` and `ClearTimeWindow` invalidating the scene cache. | Review M5: the existing fit check returns false whenever anything is outside — the normal state of a windowed chart — and a missed Clear invalidation would freeze a stale projection. | ops-test projection cases (explicit vs auto vs cleared); `pp_proj_matches_window` invariant run in every window scenario. | v2.7.1 (W1); review M5; plan §2.2 |
| SR-WIN-24 | Window commits shall be hard-ineligible for the scene-diff fast path, and PP_PROJ and PP_ROWY shall be rewritten on every window commit. | The fast path writes PP_DOC only; a stale PP_PROJ after a scale change corrupts every px↔day consumer (session-learned trap). | `pp_proj_matches_window` invariant; trace_window_commit_latency dump asserts `fastPath=false` on the commit op. | v2.7.1/v2.7.3; plan §2.2/§2.4; risk 1 |
| SR-WIN-25 | Structural-change classification shall include the window fields: `IsStructuralDocChange` and `IsStructuralDocDelta` (and the PP_DOC JSON patchers) shall treat any windowStart/windowEnd delta as structural, engaging the paint lock on window commits and never emitting a patched PP_DOC that drops a window change. | Review C3: without this, a window-only commit skips LockWindowUpdate and rewrites every prim visibly (whole-chart flash), and the patch path can silently lose the window. | ops-test patcher cases (window delta ⇒ structural / patch refused); `no_content_flash_in_trace` in trace_window_clip_rerender and trace_window_commit_latency. | v2.7.1 (W1); review C3 |

## Selection, Undo & Performance

| ID | Requirement | Rationale | Verification | Trace |
|----|-------------|-----------|--------------|-------|
| SR-WIN-26 | Any operation whose result de-emits the currently selected item — a window commit, or a nudge/drag/edit that pushes it fully outside an explicit window — shall clear ownSel to component context; edits shall never continue invisibly against a shapeless selection. | D4 + review M4 (generalized UF-07): an off-screen sticky selection invites invisible edits, phantom app-bar context, and commit-echo pops. | trace_window_clip_rerender selected-item-hidden step AND nudge-selected-task-out step (ownSel resets to component both times). | v2.7.3 (W3); D4; review M4 |
| SR-WIN-27 | A window commit shall produce exactly one undo entry (tags + shapes), Ctrl+Z shall restore the previous window and geometry, and the next operation after an undo shall not re-apply the undone window (the cached document shall be drift-checked against PP_DOC before reuse after an external undo). | SR-SMO-08 + review M2: after undoing an in-place commit the group id survives, so an unchecked scene cache silently re-applies the undone state. | trace_window_undo (commit → Ctrl+Z → window restored → nudge → window still restored, geometry coherent). | v2.7.3 (W3); review M2; SR-SMO-08 |
| SR-WIN-28 | A window commit shall reach its stable final visual state within 2000 ms (`window_commit_budget`), with a 1 s p50 target; both the in-place reconcile and delete+re-emit commit paths shall be measured before optimizing. | D3: a whole-chart view operation gets its own budget (distinct from the 200 ms single-element SR-SMO-02), but iterative window fiddling must still feel responsive; latency-truth memory says measure first. | trace_window_commit_latency (`window_commit_budget` ≤ 2000 ms via OPLATENCY; p50 recorded for both paths). | v2.7.3 (W3); D3; SR-SMO-02 |
| SR-WIN-29 | The Settings popover shall offer "Time window: Fit to tasks", which clears windowStart/windowEnd and returns the chart to auto-fit derivation, and shall display the current window while one is explicit. | Escape hatch + discoverability: users must always be able to see and reset the view state they created. | W4 settings trace/gallery capture; cold walkthrough "change-the-time-window" (walkthrough gate set). | v2.7.4 (W4); plan §1.4 |
